# 02 项目通用问题和设计

## 深入理解POJO、VO、DO的概念和应用场景

### 核心定义

| 类型 | 全称 | 主要职责 | 常见位置 |
| --- | --- | --- | --- |
| `POJO` | Plain Old Java Object | 普通对象基类概念，不绑定框架语义 | 任意层 |
| `DO` | Data Object | 与数据库表结构一一对应，用于持久化 | `infrastructure` / `mapper` |
| `VO` | View Object | 面向前端展示的数据结构 | `controller` / `api` |
| `DTO` | Data Transfer Object | 服务间或层间传输对象 | `service` / `client` |
| `BO` | Business Object | 承载业务入参和业务逻辑上下文 | `service` |

### 分层流转关系

- 控制层：接收 `VO` 或请求体对象。
- 应用层：将入参组装为 `BO` 或 `DTO`。
- 领域/服务层：执行业务规则。
- 持久层：使用 `DO` 落库与查询。
- 返回层：将 `DO` 或领域对象转换为 `VO`。

### 转换原则

- 不直接把 `DO` 暴露给前端。
- `VO` 字段按展示语义定义，不强行对齐数据库列。
- 转换逻辑集中在 `convert` 层，避免散落在控制器和服务中。

```java
public class UserConvert {

    public static UserVO toVO(UserDO userDO) {
        if (userDO == null) {
            return null;
        }

        UserVO vo = new UserVO();

        // 仅映射前端需要的字段，避免泄露内部字段
        vo.setId(userDO.getId());
        vo.setNickname(userDO.getNickname());
        vo.setStatus(userDO.getStatus());

        return vo;
    }

    public static UserDO toDO(CreateUserBO bo) {
        UserDO userDO = new UserDO();

        // 入库对象按数据库语义赋值，不混入展示逻辑
        userDO.setUsername(bo.getUsername());
        userDO.setPassword(bo.getEncodedPassword());
        userDO.setStatus(bo.getStatus());

        return userDO;
    }
}
```

### 常见反模式

- `DO` 与 `VO` 混用，导致字段污染和安全问题。
- 在控制器中直接写 `Mapper`，跳过应用层。
- 一个对象承担所有职责，最终演变为“巨型对象”。

## 深入理解类路径和物理路径的关系

### 类路径的概念

类路径（`classpath`）是 JVM 和框架在运行时查找类与资源文件的逻辑路径集合。

- `target/classes`
- `target/test-classes`
- 依赖 Jar 包

### 类路径的作用

- 加载 `.class` 文件。
- 加载配置资源，如 `application.yaml`、`mapper.xml`。
- 决定 `ClassLoader.getResource` 的查找结果。

### Java 包名与目录结构的匹配规则

| 包名 | 物理目录 |
| --- | --- |
| `cn.iocoder.yudao.module.system` | `cn/iocoder/yudao/module/system` |
| `com.example.demo.service` | `com/example/demo/service` |

规则：包名使用点分隔，物理目录使用斜杠分隔，编译后映射到 `classes` 根目录下。

### 项目、模块、包和类路径的关系

- 项目：一个代码仓库，可能包含多个 Maven/Gradle 模块。
- 模块：独立构建单元，拥有各自 `src/main/java` 与 `src/main/resources`。
- 包：源码组织单位，对应编译后的目录结构。
- 类路径：运行期聚合各模块产物与依赖，形成最终可加载空间。

```text
project-root/
  module-a/
    src/main/java/
    src/main/resources/
  module-b/
    src/main/java/
    src/main/resources/
```

### Spring 项目中的目录结构和类路径结构

```text
src/main/java      -> 编译后进入 target/classes
src/main/resources -> 原样拷贝到 target/classes
src/test/java      -> 编译后进入 target/test-classes
src/test/resources -> 拷贝到 target/test-classes
```

```java
public class ResourceDemo {

    public static void main(String[] args) {
        ClassLoader loader = Thread.currentThread().getContextClassLoader();

        // 资源路径相对 classes 根目录，不要写 src/main/resources
        var url = loader.getResource("mapper/system/UserMapper.xml");

        System.out.println(url);
    }
}
```

### 排查类路径问题的步骤

1. 确认资源文件在 `src/main/resources` 下。
2. 编译后检查 `target/classes` 是否存在该文件。
3. 检查路径是否大小写一致。
4. 多模块场景确认依赖方向是否正确。
