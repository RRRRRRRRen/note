# 07 序列化

## 深入理解Serializable接口

### 核心概念

`Serializable` 是 Java 的标记接口，用于声明对象可参与 Java 原生序列化。

- 序列化：对象转字节流。
- 反序列化：字节流还原对象。
- 未实现 `Serializable` 的对象在 `ObjectOutputStream.writeObject` 时会抛 `NotSerializableException`。

### 什么时候需要 Serializable

| 场景 | 是否需要 | 说明 |
| --- | --- | --- |
| 使用 `ObjectOutputStream` 持久化对象 | 需要 | 直接依赖 JDK 原生序列化 |
| Session/缓存底层使用 JDK 序列化器 | 需要 | 如 `JdkSerializationRedisSerializer` |
| REST 接口使用 Jackson | 不需要 | JSON 序列化与 `Serializable` 无关 |
| MQ 使用 JSON 转换器 | 不需要 | 由消息转换器处理 |

### 基本实现步骤

1. 让目标类实现 `Serializable`。
2. 显式声明 `serialVersionUID`。
3. 对敏感字段加 `transient`。
4. 在读写链路中使用 `ObjectOutputStream` / `ObjectInputStream`。
5. 类结构变化时评估兼容性影响。

```java
import java.io.Serializable;

public class User implements Serializable {

    private static final long serialVersionUID = 1L;

    private Long id;
    private String name;

    // 不参与序列化
    private transient String token;
}
```

### 最小可用读写流程

```java
import java.io.*;

public class SerializeFlowDemo {

    public static void main(String[] args) throws Exception {
        User user = new User();

        // 1) 写出对象
        try (ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream("user.bin"))) {
            out.writeObject(user);
        }

        // 2) 读回对象
        try (ObjectInputStream in = new ObjectInputStream(new FileInputStream("user.bin"))) {
            User restored = (User) in.readObject();
            System.out.println(restored);
        }
    }

    static class User implements Serializable {
        private static final long serialVersionUID = 1L;
    }
}
```

### 实践逻辑

- 先看底层是否真的使用 JDK 原生序列化，再决定是否实现 `Serializable`。
- 新项目优先使用 JSON/Protobuf，减少对 `Serializable` 的依赖。
- 若必须使用原生序列化，重点控制版本号与敏感字段。

### Spring 项目落地步骤

1. 检查 Redis/Session 是否使用 JDK 序列化器。
2. 若可调整，优先改为 JSON 序列化器。
3. 仅保留确实走 JDK 序列化链路的类实现 `Serializable`。
4. 统一异常排查：关注 `NotSerializableException`。

```java
import org.springframework.context.annotation.Bean;
import org.springframework.data.redis.connection.RedisConnectionFactory;
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.data.redis.serializer.GenericJackson2JsonRedisSerializer;

public class RedisSerializerConfig {

    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);

        // 改为 JSON 序列化，降低对 Serializable 的依赖
        template.setDefaultSerializer(new GenericJackson2JsonRedisSerializer());
        return template;
    }
}
```

### 常见问题速记

| 问题 | 结论 |
| --- | --- |
| `Serializable` 是不是“对象可传输”通用标记 | 不是，只针对 Java 原生序列化 |
| `static` 字段会被序列化吗 | 不会 |
| `transient` 字段会被序列化吗 | 不会 |
| 为什么反序列化失败 | 常见原因是类版本不兼容或引用对象不可序列化 |

### 结论

- `Serializable` 是“按需使用”的能力，不应默认加在所有类上。
- 能用 JSON/Protobuf 的场景，优先不用 JDK 原生序列化。
- 决策标准是运行链路，而不是编码习惯。

## 深入理解 @JsonIgnore

### 核心概念

`@JsonIgnore` 是 Jackson 注解，用于在 JSON 序列化和反序列化时忽略指定字段。

- 序列化时：该字段不会出现在 JSON 输出中
- 反序列化时：JSON 中的该字段会被忽略，不会赋值给对象

### 作用位置

| 位置 | 效果 |
| --- | --- |
| 字段上 | 序列化和反序列化都忽略 |
| Getter 方法上 | 仅序列化时忽略（不输出到 JSON） |
| Setter 方法上 | 仅反序列化时忽略（不从 JSON 读取） |

### 基本用法

```java
public class UserDTO {

    private Long id;
    private String username;

    @JsonIgnore  // 序列化和反序列化都忽略
    private String password;

    private String email;
}
```

序列化结果：

```json
{
  "id": 1,
  "username": "admin",
  "email": "admin@example.com"
}
```

### 单向忽略

仅序列化时忽略（允许从 JSON 读取，但不输出到 JSON）：

```java
public class UserDTO {

    private String password;

    @JsonIgnore  // 不输出到 JSON
    public String getPassword() { return password; }

    // setter 不加注解，允许反序列化时赋值
    public void setPassword(String password) { this.password = password; }
}
```

更推荐的写法是使用 `@JsonProperty(access = ...)` 精确控制：

```java
public class UserDTO {

    // WRITE_ONLY：只允许反序列化（从 JSON 读取），不序列化输出
    @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
    private String password;
}
```

### 常见使用场景

- 敏感信息：密码、token、密钥等
- 内部字段：不需要暴露给前端的业务字段
- 循环引用：避免双向关联导致的无限递归

```java
public class OrderDTO {

    private Long id;

    @JsonIgnore
    private String internalCode;  // 内部编码，不暴露

    @JsonIgnore
    private UserDTO user;  // 避免循环引用

    private BigDecimal amount;
}
```

### 与 transient 的区别

| 特性 | @JsonIgnore | transient |
| --- | --- | --- |
| 作用范围 | 仅 Jackson JSON 序列化 | JDK 原生序列化 |
| 对 JSON 的影响 | 忽略 | 无影响（仍会序列化） |
| 对 JDK 序列化的影响 | 无影响（仍会序列化） | 忽略 |

## 深入理解 @JsonInclude

### 核心概念

`@JsonInclude` 控制字段在序列化时的**包含条件**，常用于过滤 `null`、空值等不必要的字段，减少 JSON 体积。

可标注在类或字段上，字段级别的配置优先于类级别。

### Include 枚举值

| 枚举值 | 含义 |
| --- | --- |
| `ALWAYS` | 始终包含（默认行为） |
| `NON_NULL` | 非 null 时才包含 |
| `NON_EMPTY` | 非 null 且非空（空字符串、空集合等）时才包含 |
| `NON_DEFAULT` | 值不等于类型默认值时才包含（如 int 默认 0，boolean 默认 false） |
| `CUSTOM` | 自定义过滤逻辑 |

### 基本用法

字段级别：

```java
public class UserDTO {

    private Long id;

    @JsonInclude(JsonInclude.Include.NON_NULL)  // 为 null 时不输出
    private String nickname;

    @JsonInclude(JsonInclude.Include.NON_EMPTY)  // 为 null 或空集合时不输出
    private List<String> roles;
}
```

类级别（对所有字段生效）：

```java
@JsonInclude(JsonInclude.Include.NON_NULL)  // 类中所有 null 字段都不输出
public class UserDTO {

    private Long id;
    private String nickname;  // null 时不输出
    private String email;     // null 时不输出
}
```

### 全局配置

在 Spring Boot 中统一配置，避免每个类都加注解：

```java
@Configuration
public class JacksonConfig {

    @Bean
    public Jackson2ObjectMapperBuilderCustomizer customizer() {
        return builder -> builder.serializationInclusion(JsonInclude.Include.NON_NULL);
    }
}
```

或在 `application.yml` 中配置：

```yaml
spring:
  jackson:
    default-property-inclusion: non_null
```

### NON_EMPTY 的判断规则

`NON_EMPTY` 比 `NON_NULL` 更严格，以下情况都会被过滤：

- `null`
- 空字符串 `""`
- 空集合 `[]`、空 Map `{}`
- 空数组 `[]`
- `Optional.empty()`

```java
@JsonInclude(JsonInclude.Include.NON_EMPTY)
public class ResponseDTO {

    private String message;       // "" 时不输出
    private List<Object> data;    // [] 时不输出
    private Map<String, Object> extra;  // {} 时不输出
}
```

### 最佳实践

- 全局配置 `NON_NULL` 是最常见的做法，减少接口返回的冗余字段
- 需要区分 `null` 和空值语义时，在字段上单独使用 `NON_EMPTY`
- 避免在 DTO 上混用全局配置和字段级注解，容易造成行为不一致

## 深入理解 @JsonIgnoreProperties

### 核心概念

`@JsonIgnoreProperties` 是类级别的注解，用于批量忽略属性，或在反序列化时忽略未知字段。

与 `@JsonIgnore` 的区别：

| 特性 | @JsonIgnore | @JsonIgnoreProperties |
| --- | --- | --- |
| 作用位置 | 字段/方法 | 类 |
| 忽略方式 | 单个字段 | 批量指定或忽略未知字段 |
| 忽略未知字段 | 不支持 | 支持（`ignoreUnknown = true`） |

### 批量忽略指定字段

```java
@JsonIgnoreProperties({"password", "token", "internalCode"})
public class UserDTO {

    private Long id;
    private String username;
    private String password;    // 序列化和反序列化都忽略
    private String token;       // 序列化和反序列化都忽略
    private String internalCode;
}
```

### 忽略未知字段

反序列化时，如果 JSON 中包含对象没有的字段，默认会抛出 `UnrecognizedPropertyException`。使用 `ignoreUnknown = true` 可以忽略这些字段：

```java
@JsonIgnoreProperties(ignoreUnknown = true)  // 忽略 JSON 中多余的字段
public class UserDTO {

    private Long id;
    private String username;
    // JSON 中有其他字段也不会报错
}
```

全局配置（推荐）：

```java
@Configuration
public class JacksonConfig {

    @Bean
    public Jackson2ObjectMapperBuilderCustomizer customizer() {
        return builder -> builder.featuresToDisable(
            DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES  // 等同于 ignoreUnknown = true
        );
    }
}
```

或在 `application.yml` 中：

```yaml
spring:
  jackson:
    deserialization:
      fail-on-unknown-properties: false
```

### 常见使用场景

对接第三方接口时，对方返回的字段比本地 DTO 多，使用 `ignoreUnknown = true` 避免报错：

```java
@JsonIgnoreProperties(ignoreUnknown = true)
public class ThirdPartyResponseDTO {

    private String code;
    private String message;
    private Object data;
    // 第三方可能返回更多字段，不影响解析
}
```

### 最佳实践

- 全局关闭 `FAIL_ON_UNKNOWN_PROPERTIES` 比在每个类上加注解更省事
- 批量忽略字段时用 `@JsonIgnoreProperties`，单个字段用 `@JsonIgnore`，语义更清晰
- 对接外部系统的 DTO 统一加 `ignoreUnknown = true`，提高接口兼容性

## 反序列化中为什么需要空构造方法

### 核心原因

Jackson 在反序列化时需要先创建对象实例，然后再通过反射给字段赋值。默认情况下，Jackson 使用**无参构造方法**创建对象。

如果类没有无参构造方法，Jackson 会抛出异常：

```text
Cannot construct instance of `com.example.UserDTO` (no Creators, like default constructor, exist)
```

### Jackson 反序列化的对象创建流程

```java
// Jackson 的内部逻辑（简化版）
public <T> T deserialize(String json, Class<T> clazz) {
    // 1. 调用无参构造方法创建空对象
    T obj = clazz.getDeclaredConstructor().newInstance();

    // 2. 解析 JSON，通过反射给字段赋值
    for (Field field : clazz.getDeclaredFields()) {
        field.setAccessible(true);
        field.set(obj, parseValue(json, field));
    }

    return obj;
}
```

### 什么时候会缺少无参构造方法

当类显式定义了有参构造方法后，编译器不会自动生成无参构造方法：

```java
public class UserDTO {

    private Long id;
    private String username;

    // 显式定义了有参构造方法
    public UserDTO(Long id, String username) {
        this.id = id;
        this.username = username;
    }

    // 缺少无参构造方法，Jackson 反序列化会失败
}
```

反序列化时报错：

```text
com.fasterxml.jackson.databind.exc.InvalidDefinitionException:
Cannot construct instance of `UserDTO` (no Creators, like default constructor, exist)
```

### 解决方案

**方案 1：手动添加无参构造方法**

```java
public class UserDTO {

    private Long id;
    private String username;

    // 显式添加无参构造方法
    public UserDTO() {
    }

    public UserDTO(Long id, String username) {
        this.id = id;
        this.username = username;
    }
}
```

**方案 2：使用 Lombok 自动生成**

```java
@Data
@NoArgsConstructor  // 生成无参构造方法
@AllArgsConstructor // 生成全参构造方法
public class UserDTO {

    private Long id;
    private String username;
}
```

**方案 3：使用 @JsonCreator 指定构造方法**

告诉 Jackson 使用指定的构造方法进行反序列化：

```java
public class UserDTO {

    private Long id;
    private String username;

    @JsonCreator  // 指定 Jackson 使用此构造方法
    public UserDTO(@JsonProperty("id") Long id,
                   @JsonProperty("username") String username) {
        this.id = id;
        this.username = username;
    }
}
```

**方案 4：使用 @JsonDeserialize 自定义反序列化器**

适用于复杂场景，需要自定义反序列化逻辑：

```java
@JsonDeserialize(using = UserDTODeserializer.class)
public class UserDTO {

    private Long id;
    private String username;

    public UserDTO(Long id, String username) {
        this.id = id;
        this.username = username;
    }
}

public class UserDTODeserializer extends JsonDeserializer<UserDTO> {

    @Override
    public UserDTO deserialize(JsonParser p, DeserializationContext ctxt) throws IOException {
        JsonNode node = p.getCodec().readTree(p);
        Long id = node.get("id").asLong();
        String username = node.get("username").asText();
        return new UserDTO(id, username);
    }
}
```

### JDK 原生序列化的区别

JDK 原生序列化（`ObjectInputStream`）**不需要**无参构造方法，它通过 `sun.reflect.ReflectionFactory` 绕过构造方法直接创建对象。

```java
public class User implements Serializable {

    private Long id;

    // 即使只有有参构造方法，JDK 序列化也能正常工作
    public User(Long id) {
        this.id = id;
    }
}
```

### 最佳实践

- DTO/VO 类统一使用 Lombok 的 `@NoArgsConstructor` + `@AllArgsConstructor`，避免遗漏
- 如果类有业务逻辑约束（如必填字段校验），在有参构造方法中实现，无参构造方法保持空实现
- 不可变对象（`final` 字段）使用 `@JsonCreator` 指定构造方法，保持不可变性
- 避免在构造方法中执行复杂逻辑，反序列化时可能导致意外行为

### 常见问题

| 问题 | 原因 | 解决方案 |
| --- | --- | --- |
| 反序列化报 `no Creators` 错误 | 缺少无参构造方法 | 添加无参构造方法或使用 `@JsonCreator` |
| Lombok `@Data` 不生成无参构造 | 显式定义了其他构造方法 | 加上 `@NoArgsConstructor` |
| 不可变对象无法反序列化 | `final` 字段无法通过 setter 赋值 | 使用 `@JsonCreator` 指定构造方法 |

## 项目中的JSON序列化

### 序列化过程

阶段一：应用启动时（配置阶段）

```text
Spring Boot 启动
    ↓
读取 META-INF/spring/...AutoConfiguration.imports
    ↓
发现 HrJacksonAutoConfiguration
    ↓
调用 jsonUtils(List<ObjectMapper> objectMappers) Bean 方法
    ↓
创建 SimpleModule，注册所有自定义序列化器/反序列化器
    ↓
objectMappers.forEach(om -> om.registerModule(simpleModule))
    ↓
ObjectMapper 配置完成，等待使用
```

关键点：`List<ObjectMapper> objectMappers` 是 Spring 自动注入的，其中最重要的一个是 Spring MVC 内部的 ObjectMapper，它被 `MappingJackson2HttpMessageConverter` 持有。

阶段二：每次 HTTP 请求时（执行阶段）

序列化（Controller 返回数据 → JSON）：

```text
Controller 方法返回对象
    ↓
Spring MVC DispatcherServlet 处理返回值
    ↓
找到 MappingJackson2HttpMessageConverter（支持 application/json）
    ↓
调用 ObjectMapper.writeValueAsString(returnValue)
    ↓
ObjectMapper 遍历对象字段
    ↓
遇到 Long 类型 → 触发 NumberSerializer
遇到 LocalDateTime 类型 → 触发 TimestampLocalDateTimeSerializer
遇到 LocalDate 类型 → 触发 LocalDateSerializer
    ↓
写入 HTTP 响应体
```

反序列化（JSON 请求体 → Java 对象）：

```text
HTTP 请求到达（带 @RequestBody 或 @Valid 参数）
    ↓
MappingJackson2HttpMessageConverter 读取请求体
    ↓
调用 ObjectMapper.readValue(requestBody, TargetClass.class)
    ↓
遇到 LocalDateTime 字段 → 触发 TimestampLocalDateTimeDeserializer
    支持时间戳: 1710234600000 → LocalDateTime
    支持字符串: "2026-03-12 14:30:00" → LocalDateTime
    ↓
填充 Java 对象，传入 Controller 方法
```

### 触发点

```java
@RestController
@Validated
public class AttendanceDailyController {

    // 反序列化触发点：请求进来时，JSON → AttendanceDailyPageReqVO
    @GetMapping("/page")
    public CommonResult<PageResult<AttendanceDailyRespVO>> getPage(
            @Valid AttendanceDailyPageReqVO pageReqVO) {  // ← 这里触发反序列化
        // ...
    }

    // 序列化触发点：返回对象时，AttendanceDailyRespVO → JSON
    @GetMapping("/{id}")
    public CommonResult<AttendanceDailyRespVO> get(@PathVariable Long id) {
        AttendanceDailyRespVO vo = service.get(id);
        return success(vo);  // ← 这里触发序列化
    }
}
```

### 配置方法

项目中有三种配置方式，按作用范围从大到小：

#### 方式一：全局配置（修改 HrJacksonAutoConfiguration）

适合：所有接口都需要的统一规则，例如新增一个类型的全局处理。

```java
// HrJacksonAutoConfiguration.java
SimpleModule simpleModule = new SimpleModule();
simpleModule
    // 已有的配置
    .addSerializer(Long.class, NumberSerializer.INSTANCE)
    .addSerializer(LocalDateTime.class, TimestampLocalDateTimeSerializer.INSTANCE)
    .addDeserializer(LocalDateTime.class, TimestampLocalDateTimeDeserializer.INSTANCE)
    // 新增：例如 BigDecimal 保留两位小数
    .addSerializer(BigDecimal.class, new BigDecimalSerializer());
```

自定义序列化器模板：

```java
public class BigDecimalSerializer extends JsonSerializer<BigDecimal> {

    @Override
    public void serialize(BigDecimal value, JsonGenerator gen, SerializerProvider serializers)
            throws IOException {
        gen.writeString(value.setScale(2, RoundingMode.HALF_UP).toPlainString());
    }
}
```

#### 方式二：字段级配置（注解）

适合：某个字段需要特殊处理，不影响其他字段。

```java
public class EmployeeRespVO {

    // 1. 使用项目内置的序列化器
    @JsonSerialize(using = StringToArraySerializer.class)
    @JsonDeserialize(using = ArrayToStringDeserializer.class)
    private String skillIds;  // 数据库存 "[\"1\",\"2\"]"，API 返回 ["1", "2"]

    // 2. 指定日期格式（覆盖全局配置）
    @JsonFormat(pattern = "yyyy-MM-dd")
    private LocalDateTime entryDate;  // 只输出日期，不输出时间

    // 3. null 时不输出该字段
    @JsonInclude(JsonInclude.Include.NON_NULL)
    private String remark;

    // 4. 重命名字段
    @JsonProperty("emp_name")
    private String empName;

    // 5. 忽略该字段（不序列化、不反序列化）
    @JsonIgnore
    private String internalCode;
}
```

#### 方式三：application.yaml 全局属性配置

适合：调整 ObjectMapper 的行为开关，无需写代码。

```yaml
spring:
  jackson:
    # 日期格式（对未自定义序列化器的 Date 类型生效）
    date-format: yyyy-MM-dd HH:mm:ss
    time-zone: Asia/Shanghai

    # 序列化行为
    serialization:
      write-dates-as-timestamps: false  # 日期不输出时间戳
      fail-on-empty-beans: false        # 空对象不报错

    # 反序列化行为
    deserialization:
      fail-on-unknown-properties: false # 忽略 JSON 中多余的字段（推荐开启）

    # null 值处理
    default-property-inclusion: non_null  # 全局：null 字段不输出
```

注意：项目中 `LocalDateTime` 已被 `TimestampLocalDateTimeSerializer` 接管，`write-dates-as-timestamps` 对它不生效，自定义序列化器优先级更高。

### 优先级总结

```text
字段上的 @JsonSerialize/@JsonFormat  （最高，覆盖一切）
    ↓
HrJacksonAutoConfiguration 注册的 SimpleModule
    ↓
application.yaml spring.jackson.*    （最低，兜底配置）
```

### 常见场景速查

| 需求 | 配置方式 |
| --- | --- |
| 新增一个类型的全局转换规则 | `SimpleModule.addSerializer()` |
| 某个字段日期格式不同 | `@JsonFormat(pattern = "...")` |
| 某个字段不返回给前端 | `@JsonIgnore` |
| 某个字段为 null 时不输出 | `@JsonInclude(NON_NULL)` |
| 整个 VO 的 null 字段都不输出 | 类上加 `@JsonInclude(NON_NULL)` |
| 接收请求时忽略多余字段 | `yaml: fail-on-unknown-properties: false` |
| 数据库字符串 ↔ 前端数组 | `@JsonSerialize` + `@JsonDeserialize` |

## 简单描述 Easy-Trans 在项目中的使用

### 核心作用

自动将 ID 翻译为对应的名称，例如 `userId: 1` → 自动填充 `userName: "张三"`，无需手动写关联查询。

### 项目集成方式

依赖配置（`hr-spring-boot-starter-mybatis/pom.xml`）：

```xml
<dependency>
    <artifactId>easy-trans-spring-boot-starter</artifactId>  <!-- 核心 -->
</dependency>
<dependency>
    <artifactId>easy-trans-mybatis-plus-extend</artifactId>  <!-- MyBatis Plus 集成 -->
</dependency>
```

与 MyBatis Plus 集成的关键：`BaseDO` 实现了 `TransPojo` 接口：

```java
// BaseDO.java
@JsonIgnoreProperties(value = "transMap")  // Easy-Trans 会往对象里塞 transMap 字段，这里屏蔽掉避免 Jackson 报错
public abstract class BaseDO implements Serializable, TransPojo {
    // ...
}
```

`TransPojo` 是 Easy-Trans 的标记接口，告诉框架：这个类可以作为翻译的数据来源，Easy-Trans 会通过 MyBatis Plus 的 Mapper 直接查这张表。

全局配置（`application.yaml`）：

```yaml
easy-trans:
  is-enable-global: false  # 关闭全局拦截（性能考虑），改为按需在方法上加注解
```

### 使用步骤

#### 第一步：VO 类实现 VO 接口

```java
// OperateLogRespVO.java
public class OperateLogRespVO implements VO {  // ← 必须实现 VO 接口

    // 翻译配置：
    // type = SIMPLE：通过 MyBatis Plus 查 AdminUserDO 表
    // target：目标 DO 类（必须继承 BaseDO，即实现了 TransPojo）
    // fields：从目标 DO 取哪个字段的值
    // ref：翻译结果填入当前 VO 的哪个字段
    @Trans(type = TransType.SIMPLE, target = AdminUserDO.class, fields = "nickname", ref = "userName")
    private Long userId;

    private String userName;  // ← 翻译结果自动填入这里
}
```

#### 第二步：触发翻译

方式一：`@TransMethodResult` 注解（推荐，AOP 自动触发）

```java
// OperateLogController.java
@GetMapping("/page")
@TransMethodResult  // ← 方法返回后自动触发翻译
public CommonResult<PageResult<OperateLogRespVO>> pageOperateLog(...) {
    PageResult<OperateLogDO> pageResult = operateLogService.getOperateLogPage(pageReqVO);
    return success(BeanUtils.toBean(pageResult, OperateLogRespVO.class));
    // 返回前，Easy-Trans 自动扫描结果中所有 @Trans 字段并填充
}
```

方式二：`TranslateUtils.translate()` 手动触发（用于无法加注解的场景，如导出 Excel）

```java
// 导出时无法用 @TransMethodResult（返回值是 void），改为手动触发
List<OperateLogRespVO> voList = BeanUtils.toBean(list, OperateLogRespVO.class);
TranslateUtils.translate(voList);  // ← 手动触发翻译
ExcelUtils.write(response, "操作日志.xls", "数据列表", OperateLogRespVO.class, voList);
```

### 执行流程

```text
Controller 方法返回
    ↓
@TransMethodResult AOP 拦截
    ↓
扫描返回值中所有实现 VO 接口的对象
    ↓
找到 @Trans 注解的字段（如 userId）
    ↓
TransType.SIMPLE：调用 AdminUserDO 对应的 MyBatis Plus Mapper
    执行：SELECT id, nickname FROM admin_user WHERE id IN (1, 2, 3)
    ↓
将 nickname 填入 userName 字段
    ↓
返回给前端
```

### TransType.SIMPLE 为什么能自动查数据库

这就是 `easy-trans-mybatis-plus-extend` 的作用：

```text
target = AdminUserDO.class
    ↓
Easy-Trans 找到 AdminUserDO 对应的 Mapper（通过 Spring 容器）
    ↓
AdminUserDO 继承 BaseDO，BaseDO 实现 TransPojo
    ↓
Easy-Trans 知道这个 DO 有对应的 MyBatis Plus Mapper
    ↓
自动调用 selectByIds(ids) 批量查询，填充翻译结果
```

所以 `BaseDO implements TransPojo` 是集成的核心，没有这个，`TransType.SIMPLE` 就找不到数据来源。

### 常用 @Trans 参数速查

翻译单个字段：

```java
@Trans(type = TransType.SIMPLE, target = AdminUserDO.class, fields = "nickname", ref = "userName")
private Long userId;
```

翻译多个字段（同时取 `nickname` 和 `avatar`）：

```java
@Trans(type = TransType.SIMPLE, target = AdminUserDO.class,
       fields = {"nickname", "avatar"}, ref = {"userName", "userAvatar"})
private Long userId;
```

跨模块时用 `targetClassName`（避免循环依赖）：

```java
@Trans(type = TransType.SIMPLE, targetClassName = "com.hsf.hr.module.system.dal.dataobject.user.AdminUserDO",
       fields = "nickname", ref = "userName")
private Long userId;
```
